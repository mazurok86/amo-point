# Этап 2 — JS: динамическая видимость полей по типу

Решение НЕ интегрируется в Laravel-приложение — это standalone-файл/сниппет под чужую страницу `https://test.amopoint-dev.ru/testzz/testlist.html` (исходный URL отвечает 301 → HTTPS).

## Реальная структура целевой страницы (curl)

```html
<select name="type_val">
    <option value="1">1</option> ... <option value="5">5</option>
</select>

<p>Поле N   <input name="input_N" type="text"></p>   (N = 1..7)
<p><input name="button_12" type="button" value="Кнопка 1"></p>
<p><input name="button_28" type="button" value="Кнопка 2"></p>
<p><input name="button_88" type="button" value="Кнопка 4"></p>
<p><input name="button_33" type="button" value="Кнопка 3"></p>
<p><input name="button_1"  type="button" value="Кнопка 8"></p>
```

Ключевое:
- Селект называется `type_val`, а не `type` — в селекторе используем `select[name*="type" i]`, чтобы не зависеть от точного имени.
- Кнопки `<input type="button" name="button_*">` — тоже «поля с name», по ТЗ должны участвовать в фильтре.
- Каждое поле обёрнуто в `<p>` (с инлайн-текстовой меткой «Поле N»). `closest('p')` корректно скрывает обёртку вместе с меткой.

## Поставка

- **`public/testzz/testlist.js`** — основной деливерабл, IIFE без зависимостей. Подключается тегом `<script src="testlist.js">`, либо тело IIFE копируется в DevTools Console.
- **`public/testzz/testlist.html`** — локальный полигон. Доступен по `http://amo-point.local/testzz/testlist.html` (vhost `amo-point.local` отдаёт `public/` напрямую через `try_files`). Содержит варианты разметки: `<label>+input` оборачивающий, `<label for=>` отдельный, `<fieldset>`, текстарея и второй `<select>`.

## Алгоритм (выбранный)

1. На `DOMContentLoaded` (или сразу, если DOM уже готов — для сценария «вставить в Console») находим селект `select[name="type_val"]` — на целевой странице это конкретный единственный элемент.
2. Навешиваем `change` напрямую на найденный select — страница статичная, делегирование смысла не имеет.
3. На каждое срабатывание:
   - `value = select.value.trim().toLowerCase()`;
   - перебираем `input[name]` — на целевой странице все фильтруемые поля это `<input>` (тексты + кнопки). Source-`<select>` под селектор не подпадает, отдельная проверка `field === sourceSelect` не нужна;
   - `match = field.name.toLowerCase().includes(value)`;
   - **скрываем/показываем родительский элемент** (`field.parentElement`) — на целевой странице это `<p>`, который содержит и input, и инлайн-метку «Поле N»/«Кнопка N». Один доступ, без `closest`-обхода.
4. Запускаем один раз сразу после init — чтобы начальное состояние было консистентно (на целевой странице select по умолчанию `value="1"`, поэтому сразу скрываются поля без «1» в имени).
5. Если `value === ''` — показываем все поля (sane default; на целевой странице такого варианта в options нет, но сохраняем для устойчивости).

## Что считать «содержит»

Приоритетный вариант — `String.prototype.includes` (substring). Почему:
- ТЗ дословно говорит «есть значение элемента списка» → подстрока, не точное совпадение.
- Покрывает кейсы вида `phone_mobile`, `phone_home`, где `value="phone"`.

Альтернатива — `name.split(/[_\-.]/).includes(value)` (по разделителям). Точнее, но рискует не совпасть при склейках без разделителей. Указать в README.

## Альтернативы (с обоснованием отказа)

| Подход | Почему не выбран |
|---|---|
| **CSS attribute selector** + динамический `<style>`: `input[name*="x"] { display: revert } ; input[name] { display: none }` | Лаконично, но прячет только сам input, не родительский label/контейнер. Селектор `:has()` решает, но поддержка хуже. |
| **Универсальный `closest('.form-row, .field, fieldset, p, label, li')` + `label[for=]` через `CSS.escape`** | Покрывает разные формы разметки (вложенный label, отдельный label через `for=`, fieldset и т.д.), но это спекулятивная сложность под кейсы вне ТЗ. На целевой странице обёртка — всегда `<p>` (= `parentElement`), доступ в одну строку без обходов. Если потребуется обобщить — вернуть resolver несложно. |
| **Расширенный `FIELD_SELECTOR` (`input[name]:not(hidden|submit|reset), textarea[name], select[name]` + ручное исключение source-select по reference)** | Целевая страница содержит только `<input>` (тексты и кнопки). Расширенный селектор — задел под формы, которых в ТЗ нет. С простым `input[name]` source-`<select>` сам по себе не попадает в выборку — ещё одна развилка отпадает. |
| **Делегирование `change` на `document` с фильтром `e.target.matches(SELECT_SELECTOR)`** | Полезно если select добавляется/пересоздаётся динамически. Целевая страница статичная — прямое `select.addEventListener('change', ...)` короче и не требует matching на каждое событие в документе. |
| **Гибкий селектор `select[name*="type" i]` (case-insensitive substring) вместо точного `[name="type_val"]`** | Защищает от переименования, но это защита от ситуации, не описанной в ТЗ. На реальной странице select ровно один и точно с этим именем — точечный селектор очевиднее в коде. Если страница изменится — поменять одну строку. |
| **MutationObserver** на форму | Нужен только при динамическом добавлении полей. На статической форме оверкилл. |
| **jQuery** `$('input[name*=...]').show()` | Лишние ~30 КБ ради десятка строк нативного JS. |
| **Alpine / Vue / React** | Требует переписывать разметку формы. ТЗ просит подключаемый сниппет к существующей странице. |
| **Жёсткая мапа `{type: [имена полей]}`** | Не масштабируется, противоречит ТЗ («поля, в name которых есть значение»). |
| **Скрытие через `hidden` атрибут vs `style.display='none'`** | Использую `hidden` — семантично, не конфликтует с inline-стилями страницы. |
| **Анти-FOUC через инжект `<style>` с `:has()` + удаление после `init()`** | Не делаем: ТЗ не требует, в основном сценарии (Console-сниппет на чужой странице) проблема не возникает в принципе, а для `<script src>` в `<head>` и для маленькой целевой страницы реального мерцания не наблюдается. ~40 строк ради граничного случая размывают ядро алгоритма, который и проверяют. |
